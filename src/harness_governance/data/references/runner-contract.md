# Runner Contract / Runner 契约

## Purpose / 目的

The runner is a native subagent handoff coordinator. It prepares prompt/request files, records the platform-native spawn that the host agent performs, parses the subagent result, and leaves a gateable lifecycle trail. It does not directly execute platform CLIs or subprocess agent workers.

`autonomous-ready-loop` 通过一次只运行一个可丢弃的 agent worker，避免上下文窗口持续膨胀。runner 负责重复调度，仓库文件负责记忆，模型只负责一次有边界的执行尝试。

## Roles / 角色

- **Runner**: Prepares native handoff requests, records native spawn metadata, parses subagent result JSON, and gates the lifecycle. It does not launch external agent worker processes.
- **Runner**：准备 native handoff request，记录平台原生 spawn 元数据，解析 subagent 结果 JSON，并对生命周期进行 gate；它不启动外部 agent worker 进程。
- **Worker**: Reads repository instructions, executes one ready item or a limited batch, updates persistent files, runs verification, and outputs a marker.
- **Worker**：读取仓库说明，执行一个 ready 项或有限批次，更新持久化文件，运行验证，并输出 marker。
- **Queue file**: Defines the next executable item and phase boundary. Examples: `NEXT.md`, `TODO.md`, issue export, or backlog JSON. `NEXT.md` should serve as the scheduler, keeping only executable `[ready]` items and briefly `[active]` when necessary.
- **Queue file**：定义下一个可执行项和阶段边界。例如 `NEXT.md`、`TODO.md`、issue export 或 backlog JSON。`NEXT.md` 应作为 scheduler，只保留可执行 `[ready]` 和必要时短暂 `[active]`。
- **Done archive**: Stores completed change packets or done history, defaulting to `docs/changes/archive/<YYYY-MM-DD>-<change-id>/`; completed records should not remain in the scheduler long-term.
- **Done archive**：保存已完成 change packet 或 done 历史，默认 `docs/changes/archive/<YYYY-MM-DD>-<change-id>/`；完成记录不要长期留在 scheduler。
- **Checkpoint**: Records what the last worker did and how the next worker can continue without relying on chat history.
- **Checkpoint**：记录上一个 worker 做了什么，以及下一个 worker 如何不依赖聊天历史继续。
- **Native handoff lifecycle log**: Records prepared request, prompt hash, native agent id, parse completion, verdict, verification status, and findings summary for every governed subagent handoff.
- **Native handoff lifecycle log**：记录 prepared request、prompt hash、native agent id、parse completion、verdict、verification status 和 findings summary。
- **Status output**: Generated by the common `harness-visualization` as `.harness/status.md` and `.harness/status.json`, for humans and subsequent agents to view the current layer, scheduler queue, done archive, task packet, runner, and verification status.
- **Status output**：由通用 `harness-visualization` 生成 `.harness/status.md` 和 `.harness/status.json`，供人类和后续 agent 查看当前 layer、scheduler queue、done archive、task packet、runner 和 verification 状态。

## Run Modes / 运行模式

### BoundedBatch

The default safe mode. The runner runs at most `MaxRounds` rounds, suitable for initial verification, CI smoke tests, or low-risk batch processing. Starting with `MaxRounds=1` is recommended.

默认安全模式。runner 最多运行 `MaxRounds` 轮，适合初次验证、CI smoke test 或低风险批处理。推荐从 `MaxRounds=1` 开始。

### RunUntilBoundary

The target autonomous mode. The runner keeps launching new workers until a project boundary or safe-stop condition is reached. In this mode, `MaxRounds` is a fuse, not a task goal; common upper limits can be `50` or `100`.

目标自治模式。runner 持续启动新 worker，直到达到项目边界或安全停止条件。此模式下 `MaxRounds` 是保险丝，不是任务目标；常见上限可以是 `50` 或 `100`。

When using `RunUntilBoundary`, the worker still processes only one ready item or an explicitly allowed limited batch at a time. Context cleanup is achieved by "new worker"; task continuity is achieved by queue files and checkpoints.

使用 `RunUntilBoundary` 时，worker 仍然一次只处理一个 ready 项或明确允许的有限批次。上下文清理由"新 worker"实现，任务连续性由队列文件和 checkpoint 实现。

## Minimum Checkpoint / 最小 Checkpoint

Use the project's own path, e.g. `.harness/run-checkpoint.md` or `agents/run-checkpoint.md`.

使用项目自己的路径，例如 `.harness/run-checkpoint.md` 或 `agents/run-checkpoint.md`。

```markdown
# Run Checkpoint

## Last Worker
- Started: <timestamp>
- Queue item: <id/title>
- Result: done | handoff | blocked | boundary | failed

## Durable State Updated
- <files or artifacts updated>

## Verification
- `<command>` -> pass | fail | not run, with reason

## Next Resume Source
- Queue: <path>
- First ready item: <id/title or none>

## Stop Reason
- <blank unless blocked/boundary/failed>
```

The runner must persist all agent invocation records, not just print the latest output to the terminal.

runner 必须把所有 agent 调用记录持久化，不只是在终端打印最近一次输出。

Suggested paths:

建议路径：

- `.harness/native-handoffs/<session>.ndjson`: One structured native handoff lifecycle record per line.
- `.harness/native-handoffs/<session>.ndjson`：每行一条结构化 native handoff lifecycle 记录。
- `.harness/subagent-requests/<session>/<request-id>.json`: Prepared native handoff request metadata.
- `.harness/subagent-requests/<session>/<request-id>.json`：已准备的 native handoff request 元数据。
- `.harness/tmp/<role>-prompt.md`: Prepared prompt for host-platform native subagent spawning.
- `.harness/tmp/<role>-prompt.md`：供宿主平台原生 subagent spawn 使用的 prompt。
- `.harness/status.md`: Human-readable status dashboard.
- `.harness/status.md`：人类可读状态仪表。
- `.harness/status.json`: Machine status for agents, TUI, or Web UI consumption.
- `.harness/status.json`：agent、TUI 或 Web UI 消费的机器状态。
- `.harness/harness-status.config.json`: Only needed when project paths do not follow default conventions.
- `.harness/harness-status.config.json`：仅当项目路径不符合默认约定时才需要。
- `docs/changes/archive/`: Archive directory for completed change packets; old `[done]` queue items should preserve history and verification evidence when migrated.
- `docs/changes/archive/`：完成后的 change packet 归档目录；旧 `[done]` 队列项迁移时应先保留历史和验证证据。

每条 invocation log 至少包含：

- startedAt / finishedAt。
- round.
- round。
- queue item.
- queue item。
- requestId and native agent id.
- requestId 和 native agent id。
- prompt hash.
- prompt hash。
- lifecycle status.
- lifecycle status。
- prompt and request file paths.
- prompt 和 request 文件路径。
- verification summary.
- verification summary。

- commit hash 或 commit skipped reason。

If the user requests "output all invocation records", the runner should read the invocation log and summarize all rounds, not just display the last round.

如果用户要求"输出所有调用记录"，runner 应读取 invocation log 并汇总所有轮次，而不是只展示最后一轮。

## Status Refresh Requirements / 状态刷新要求

The runner template should automatically call `harness status --refresh` at the following times:

runner 模板应在以下时机自动调用 `harness status --refresh`：

- After a worker completes normally and finishes matching verification.
- worker 正常完成并跑完匹配 verification 后。
- After the runner writes a timeout, failed, blocked, boundary, or unrecognized marker checkpoint.
- runner 写入 timeout、failed、blocked、boundary 或 unrecognized marker checkpoint 后。
- When an external runner initializes project paths, it may first run `--init` to generate `.harness/harness-status.config.json`, but projects using default paths do not need configuration.
- 外部 runner 初始化项目路径时，可先运行 `--init` 生成 `.harness/harness-status.config.json`，但默认路径项目不需要配置。

Business projects should not copy visualization logic. Business projects are only responsible for maintaining the scheduler queue, done archive, change packets, checkpoints, invocation logs, and necessary verification facts; stable verification records are only needed for cross-round recovery, status changes, release/PR, verification mechanism changes, failure/skipped, or residual risk.

业务项目不应复制可视化逻辑。业务项目只负责维护 scheduler queue、done archive、change packet、checkpoint、invocation log 和必要 verification 事实；stable verification record 仅在跨轮恢复、状态变更、release/PR、验证机制变更、失败/skipped 或 residual risk 时需要。

## Verification Preset Requirements / Verification preset 要求

The runner template must only execute controlled verification presets; it must not pass arbitrary shell strings from queue text, config text, or environment variables directly to `eval`, `Invoke-Expression`, or equivalent execution entry points.

runner 模板只能执行受控 verification preset，不应把 queue 文本、config 文本或环境变量中的任意 shell 字符串直接传给 `eval`、`Invoke-Expression` 或等价执行入口。

Current common presets:

当前通用预设：

- `routing-guardrails`: Runs `harness check routing`.
- `routing-guardrails`：运行 `harness check routing`。
- `all-local-checks`: Runs `harness check all`.
- `all-local-checks`：运行 `harness check all`。
- `all-local-checks`: Chains all local checks (routing, packets, entry, inventory).
- `all-local-checks`：串联所有本地检查（routing、packets、entry、inventory）。

If a project needs additional verification commands, they should first be registered as new runner presets and updated in both the runner asset and corresponding tests. `VerificationCommand` must only be used as a preset name, not as an unaudited command-line passthrough.

如果项目需要更多验证命令，应先把它们登记成新的 runner preset，并在 runner asset 和对应测试中同步更新。`VerificationCommand` 只能作为 preset 名称使用，不能作为未审计命令行透传。

## Worker Prompt Requirements / Worker Prompt 要求

Project-specific worker prompts should include the following constraints:

项目专用 worker prompt 应包含以下约束：

```text
只从仓库文件恢复状态。不要依赖上一轮聊天。
读取项目说明和队列来源。
只执行第一个 ready 项，除非配置明确允许有限批次。
保持改动在 ready 项边界内。
如果 ready 项进入实现代码，先运行 implementation-readiness-gate。
运行匹配验证。
退出前更新 scheduler queue、done archive、checkpoint；只有跨轮恢复、状态变更、失败/skipped/风险或验证机制变更需要未来复用时，才更新 stable verification record。
runner 会在 worker 退出后刷新 `.harness/status.md` 和 `.harness/status.json`。
只有项目规则要求自动 commit 且验证通过时，才 commit。
如果 context 看起来接近 70%，写 checkpoint 并输出 AUTONOMOUS_CONTEXT_HANDOFF。
最终回复必须且只能包含一个 AUTONOMOUS_* marker。
```

## Runner Safety Defaults / Runner 安全默认值

- Use the host platform's native subagent capability. The CLI may prepare, record, parse, and gate the handoff, but it must not pretend to spawn native subagents itself.
- 使用宿主平台的原生 subagent 能力。CLI 可以 prepare、record、parse、gate handoff，但不能假装自己直接 spawn 原生 subagent。
- Prefer sandbox mode (e.g. `--sandbox workspace-write`).
- 优先使用沙箱模式（如 `--sandbox workspace-write`）。
- Do not pass approval flags from interactive entry to non-interactive workers; workers should rely on sandboxing and failure-exit mechanisms.
- 不要把交互入口的审批标志传给非交互 worker；worker 应依赖沙箱和失败退出机制。
- Do not use flags that bypass approval and sandboxing unless the external environment is separately sandboxed and the user explicitly requests it.
- 不要使用绕过审批和沙箱的标志，除非外部环境已经单独沙箱化，且用户明确要求。
- Default to `BoundedBatch` with a small `MaxRounds`; use `RunUntilBoundary` explicitly when the user's goal is to automatically complete the current phase.
- 默认使用 `BoundedBatch` 和小 `MaxRounds`；当用户目标是自动完成当前阶段时，显式使用 `RunUntilBoundary`。
- Write the worker's final message to a file and parse the marker from that file.
- 把 worker 的最终消息写到文件，并从该文件解析 marker。

## Continuation Conditions / 继续条件

The runner may continue launching the next round only when all of the following conditions are met:

只有同时满足以下条件时，runner 才能继续启动下一轮：

- The worker exited successfully.
- worker 成功退出。
- The final marker is `AUTONOMOUS_READY_DONE` or `AUTONOMOUS_CONTEXT_HANDOFF`.
- 最终 marker 是 `AUTONOMOUS_READY_DONE` 或 `AUTONOMOUS_CONTEXT_HANDOFF`。
- The queue still has ready items.
- 队列中仍有 ready 项。
- Required verification commands pass, or the project explicitly allows the worker to continue fixing in the next round and the failure threshold has not been exceeded.
- 必需验证命令通过，或项目明确允许 worker 在下一轮继续修复且未超过失败阈值。
- No boundary violation or human input need.
- 没有边界越界或人工输入需求。
- The `MaxRounds` safety limit has not been reached.
- 尚未达到 `MaxRounds` 安全上限。

Otherwise the runner must stop and report the reason.

否则必须停止并报告原因。

## Scope Budget / 范围预算

After each round, the runner checks the git diff against scope budget thresholds. When exceeded, it forces truncation and automatically splits the task to prevent agent drift and hallucination.

runner 在每轮执行后检查 git diff，与 scope budget 阈值比较。超限时强制截断并自动拆分任务，防止 agent 漂移和幻觉。

### Three-Layer Coordination / 三层联动

1. **Definition-time constraint**: The `[ready]` block in NEXT.md declares the task granularity upper bound via the `Scope:` field.
2. **定义时约束**：NEXT.md 的 `[ready]` 块通过 `Scope:` 字段声明任务粒度上限。
3. **Execution-time fallback**: After each round, the runner checks the git diff; if exceeded, `stop_for=scope_exceeded`.
4. **执行时兜底**：runner 每轮执行后检查 git diff，超限则 `stop_for=scope_exceeded`。
5. **Auto-recovery on exceedance**: The original task is marked `[blocked]`, and split `[ready]` sub-tasks are automatically generated.
6. **超限自动恢复**：原任务标记为 `[blocked]`，自动生成拆分后的 `[ready]` 子任务。

### Scope Field Format / Scope 字段格式

Use the `Scope:` field in the `[ready]` block of NEXT.md:

在 NEXT.md 的 `[ready]` 块中使用 `Scope:` 字段：

```markdown
[ready] Fix login bug
- Layer: implementation
- Scope: 5/300
```

**Short format**: `N/M` = max N files / M diff lines

**短格式**：`N/M` = 最多 N 个文件 / M 行 diff

```markdown
- Scope: 5/300
```

**Long format**: comma-separated `key=value` pairs

**长格式**：`key=value` 逗号分隔

```markdown
- Scope: max-files=5, max-diff-lines=300, forbidden=src/core/**, owner=src/auth.py
```

| Parameter | Meaning | Default |
|-----------|---------|---------|
| `max-files` | Maximum number of files to modify | 0 (no limit) |
| `max-diff-lines` | Maximum number of added lines | 0 (no limit) |
| `forbidden` | Paths that must not be touched (glob; semicolon-separated for multiple) | None |
| `owner` | Declare only these files may be touched (glob; semicolon-separated for multiple) | None |

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `max-files` | 最多修改文件数 | 0（不限制） |
| `max-diff-lines` | 最多新增行数 | 0（不限制） |
| `forbidden` | 禁止碰的路径（glob，分号分隔多个） | 无 |
| `owner` | 声明只动这些文件（glob，分号分隔多个） | 无 |

### Configuration / 配置方式

Priority from highest to lowest:

优先级从高到低：

1. **NEXT.md per-task**: `Scope: 3/200` (constrains only that task)
2. **NEXT.md 单任务**：`Scope: 3/200`（只约束该任务）
3. **Configured fallback**: project-level `scope_budget = "5/300"` applies to runner/gate checks that inspect changed files.
4. **CLI scope override removed**：使用项目配置和 native handoff lifecycle gate。
5. **config.toml project default**: `scope_budget = "10/800"`
6. **config.toml 项目默认**：`scope_budget = "10/800"`
7. **No configuration**: scope checking disabled
8. **无配置**：scope 检查禁用

### Recommended Values / 推荐值

| Scenario | max_files | max_diff_lines |
|----------|-----------|----------------|
| Strict (prevent drift) | 3-5 | 150-300 |
| Standard (default) | 10 | 800 |
| Loose (large refactor) | 20 | 2000 |
| Disabled | 0 | 0 |

| 场景 | max_files | max_diff_lines |
|------|-----------|----------------|
| 严格（防漂移） | 3-5 | 150-300 |
| 标准（默认） | 10 | 800 |
| 宽松（大重构） | 20 | 2000 |
| 禁用 | 0 | 0 |

### Exceedance Behavior / 超限行为

When the scope budget is exceeded:

当 scope budget 被突破时：

1. The runner stops the current round with `stop_for=scope_exceeded`.
2. runner 停止当前轮次，`stop_for=scope_exceeded`。
3. A checkpoint is written recording the exceedance details.
4. 写入 checkpoint 记录超限详情。
5. The original task is automatically marked `[blocked]`, and split `[ready]` sub-tasks are generated (default `max-files=3, max-diff-lines=200`).
6. 自动将原任务标记为 `[blocked]`，并生成拆分后的 `[ready]` 子任务（默认 `max-files=3, max-diff-lines=200`）。
7. The exit code is 1 (non-zero).
8. 退出码为 1（非零）。
