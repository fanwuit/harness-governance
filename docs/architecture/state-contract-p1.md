# Architecture: State Contract P1 Enhancements

## Boundaries
- `commands/state_contract.py` remains the source of state-contract requirements.
- `commands/check.py` owns `CheckResult` adapters and `harness check all`.
- `state_machine/gates.py` owns verification-layer gate hooks.
- `commands/init.py` owns project scaffolding generated by `harness init`.

## Component Responsibilities
- State-contract command: enumerate writer -> consumer requirements and direct
  CLI output for `harness state-contract check`.
- Check command: adapt the same requirements into standard check findings.
- Gate engine: call the check adapter during verification and surface failures.
- Init command: create a downstream scaffold under `tests/` so initialized
  projects inherit the state-contract testing expectation.

## Owners
- Harness CLI command layer owns user-facing check commands.
- Harness state machine owns gate enforcement.
- Harness init scaffolding owns generated project files.

## Data Flow
`_REQUIREMENTS` in `state_contract.py`
-> `check_state_contract(project_root)` in `check.py`
-> `harness check state-contract`, `harness check all`, and verification hook
-> user-visible pass/fail output and gate failure messages.

`harness init`
-> writes project config, skills, queue/docs scaffolding
-> writes `tests/test_state_contract_scaffold.py`
-> downstream project has a concrete place to replace scaffold assertions with
   real writer -> consumer tests.

## ADR Candidates
- No standalone ADR is required. This is an extension of the existing state
  contract closure decision, not a new long-lived architecture boundary.
