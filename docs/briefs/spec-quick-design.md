# Design: Spec Quick — Lightweight Spec Mode

## Goal

Define a single-file lightweight spec mode (`harness spec quick`) as an
alternative to the full 5-file change packet, and specify the upgrade
boundary between them.

## Motivation

The full change packet requires 5 files:
- `proposal.md`, `contracts.md`, `design.md`, `tasks.md`, `verification.md`

For simple tasks (single-file change, config tweak, small bug fix), creating
5 files is disproportionate friction. But "just describe it" with no structure
loses traceability. Spec quick fills the gap.

## Spec Quick Format

A single markdown file at `.harness/specs/<slug>.md` with these sections:

```markdown
# Spec: <title>

## Goal
One-sentence summary of what this achieves.

## Scope
- In scope:
- Out of scope:

## Contract Delta
What observable behaviour changes. Reference existing contracts/tests.

## Tasks
- [ ] Task 1
- [ ] Task 2

## Verification
How to verify this works.

## Upgrade Notes
Why this is spec quick and not a full packet (optional, auto-generated).
```

Generated by:
```bash
harness spec quick "add dark mode toggle"
```

Which creates `.harness/specs/add-dark-mode-toggle.md` with the template above.

## Upgrade Boundary Criteria

Upgrade from spec quick → full change packet when **any** of:

| Criteria | Rationale |
|---|---|
| Affects 3+ files | Complexity exceeds single-spec scope |
| Touches public API/CLI/contract | Requires contract review |
| Has external side effects | Needs ADR + architecture |
| Involves 2+ engineers | Needs task breakdown + ownership |
| Persists new state | Needs state-contract evidence |
| Takes >1 session to implement | Needs task tracking |
| Requires dependency change | Needs design review |
| Author explicitly requests packet | User preference |

**Default**: spec quick for trivial-safe-change and fast-path governed tasks;
full change packet for governed-path tasks that meet any upgrade criterion.

## CLI Design

```bash
harness spec quick <description> [--path <path>]
  # Create a lightweight spec file from description

harness spec upgrade <spec-path> [--change-id <id>]
  # Upgrade a spec quick to a full change packet:
  # 1. Read spec quick fields
  # 2. Generate 5 packet files with pre-filled content
  # 3. Create docs/changes/<id>/ with the packet files

harness spec list
  # List all specs in .harness/specs/
```

## Implementation Plan (P2 — deferred)

2. Create command group `src/harness_governance/commands/spec.py`
3. Implement `spec quick` — generate single file from template
4. Implement `spec upgrade` — promote spec to full change packet
5. Implement `spec list` — enumerate existing specs
6. Register in `cli.py`
7. Tests in `tests/test_commands/test_spec_cmd.py`

## Non-Goals

- Spec quick does NOT replace gate checks; it is a documentation carrier.
- Spec quick does NOT auto-approve implementation; readiness gate still
  requires Implementation Entry Record.
- Spec quick does NOT support ADR/architecture sections; upgrade to
  full packet when those are needed.
