### 原则：Superpowers 吸收边界（不计入整改优先级）

- 适用范围：以下所有 Superpowers 吸收项。
- 已完成：`harness-engineering/references/superpowers-routing.md` 已把 Superpowers 定位为 companion-only，并列出可借用场景。
- 已完成：已新增 Local Absorption Map，并把各借用动作落到本地 owner skill 的 reference/template。
- 边界：只吸收 Superpowers 的工程动作，不吸收它的流程主权。每个吸收项都必须写清楚：
  - Local owner：由哪个本地 governance skill 拥有。
  - Allowed technique：允许借用哪种 Superpowers 技巧。
  - Forbidden transition：禁止执行哪些 terminal state、next skill、默认 artifact、commit / branch 动作。
  - Stable evidence：结果必须写入哪个本地证据，例如 brief、contract、test、verification、review-next、change packet 或 queue。
  - Verification：如何证明该技巧没有绕过 harness layer。
- 验收标准：新增或更新的文档中必须出现“companion-only / 不抢入口 / 不批准 implementation”的边界说明。

### P1：吸收 systematic debugging 的根因追踪优势

- 状态：已完成。
- 背景：Superpowers 的 systematic debugging 强在“先复现、读错误、追数据流、对照工作样例、最小假设验证”；本项目目前的 `debugging-checklist` 更偏人类 handoff。
- 已完成：新增 `observable-fact-discovery/references/debugging-fact-workflow.md`，并让 `debugging-checklist` 明确只作为轻量 handoff：
  - Reproduce：记录稳定复现步骤或无法复现的证据。
  - Observe：记录错误、日志、输入输出、环境。
  - Isolate：定位最小失败边界和工作对照样例。
  - Hypothesis：一次只验证一个假设。
  - Fix / Verify：修复后回到 verification 和 review-next。
- 禁止：不能先猜修；不能用截图或主观判断替代可复现事实；发现未知行为时回到 Fact Discovery。
- 验收：模板能让 future agent 在 bug/test failure 上先产出事实证据，再进入 implementation。

- 验收：最终汇报和 review-next 记录都能追溯到新鲜命令输出或明确风险。
