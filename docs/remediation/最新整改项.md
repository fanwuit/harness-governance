# 最新整改项

生成日期：2026-06-12

## 背景

本轮整改在原 `剩余整改项.md` 的 4 个优先收尾项基础上，追加用户反馈的治理链路降频要求：部分检查、文档园丁和 verification record 不应被理解为每次实现完成都必须全量触发。目标是保持治理证据可追溯，同时降低日常迭代开销。

## 总体原则

- 迭代中跑 targeted checks；阶段收口、commit / PR / push / release 前跑 `npm run check:all`。
- 周期性文档园丁默认 scan-only；repair mode 需要低频或人工批准。
- 完成前必须有 verification evidence；只有跨会话、状态变更、PR / release、验证机制改动、失败 / skipped / residual risk 等需要未来复用的场景，才写 stable verification record。
- 新增治理能力必须落到本地 owner skill，不引入外部流程主权或 adapter。

1. 先落地降频边界：检查频率、文档园丁 scan-only / repair、verification evidence vs stable record。

3. 再做 status contract 去重与 QA / Release / Monitor / Retro 本地吸收。
4. 最后更新 `剩余整改项.md`，并运行 targeted checks 与一次 `npm run check:all`。
